# Runtime AI Evaluation Harness

Evaluate the **runtime narrator** â€” the prompts + models that generate in-game
narration â€” by feeding one fully-specified interaction event to a model and
scoring the response with pluggable judges.

This is the third evaluation harness in the repo, alongside the blueprint
pipeline (`evaluation/`) and the game-master trace pipeline (`evaluation/trace/`).
Those grade authored blueprints and finished sessions; this one grades a single
runtime narration event in isolation.

## Determinism is the point

A **case evaluates exactly ONE action** against a **fully-specified prior state**
â€” the complete, fixed conversation history is part of the case, not generated by
playing turns. So the model input is identical every run and **identical across
models**: swap the backend or the model and you are comparing apples to apples on
the same input. (An earlier multi-turn design let each model see its *own* prior
narration, so models diverged as turns accumulated and were not comparable.)

## Concepts

- **Case** â€” `cases/*.mjs`, default-exporting one case or an array of cases:
  ```js
  {
    id: "ask-alice-alibi-pressure",
    blueprint: { path: "supabase/seed/blueprints/mock-blueprint.json" }, // target_age 10
    given: {                          // the EXACT fixed prior state
      mode: "talk",                   // explore | talk | accuse
      location_id: "loc-kitchen",
      talk_character_id: "char-alice",
      time_remaining: 7,
      history: [                      // complete prior conversation, deterministic
        { event_type: "ask", actor: "system",
          narration: "\"I never went near the jar,\" Alice says.",
          payload: { character_id: "char-alice", player_input: "Did you take the cookies?" } },
      ],
    },
    action: { type: "ask", player_input: "Then why are your hands shaking?" }, // the ONE event
    judges: ["flesch"],
    judgeConfig: { flesch: { tolerance: 2 } },
  }
  ```
  `history` rows mirror how the runtime stores `game_events`
  (`event_type`/`actor`/`narration`/`payload`), so the **same data drives both
  backends** â€” the endpoint backend seeds them as rows; the CLI backend feeds
  them to the context builder. Supported `action.type`s and the state each is
  valid from are defined in `lib/roles.mjs`.

- **Model backend** â€” produces the response. One interface
  (`collect(testCase, ctx) -> { response, blueprint, model }`):
  - `endpoint` â€” seeds the fixed session + history into the DB (service role),
    then calls the ONE `game-*` endpoint. The server rebuilds the exact context
    from the seeded rows, so the only variable is the model behind the session's
    `ai_profile` (`mock` for plumbing/CI; an `openrouter` profile reaching
    `openai/*` or `anthropic/*` for a real run). Supports every action.
  - `cli:<variant>` â€” reconstructs the **real** runtime prompt+context for the
    action (reusing the shared builders in `supabase/functions/_shared`, imported
    directly â€” Node strips the TypeScript types) and pipes it to a local CLI
    (`claude`, `openai`, or a deterministic `stub`). No game server needed.
    Replays the roles with a `cli` mapping in `lib/roles.mjs` (`talk`, `ask`
    today); add a mapping to support another.

- **Judge** â€” scores a stored interaction's single response
  (`judge(interaction, { config }) -> { id, status, score, details, parts }`).
  First judge: `flesch`, a deterministic Fleschâ€“Kincaid grade-level check
  against the blueprint's `target_age`. Add judges in `lib/judges/`.

- **Collect â†’ judge split** â€” every run writes the raw `interaction.json`
  **before** any judge runs, then judges into `result.json`. Re-run judges over a
  stored interaction with `rejudge.mjs` â€” no endpoint or model calls. This is the
  inner loop while iterating on judges.

## Artifacts (`runs/<run_id>/<case>__<backend>/`, gitignored)

- `interaction.json` â€” raw capture: the `given` state, the `action`, the
  `response` (narration + rendered prompt for the CLI backend), and
  case/backend/model metadata. The re-judgeable source of truth.
- `result.json` â€” judges over that interaction, with a pass/fail summary.
- A run-level `runs/<run_id>/summary.json` lists every `(case Ă— backend)` result.

## Running

The `endpoint` backend needs the local Supabase stack up and seeded:

```bash
npm run supabase:restart   # start the worktree's stack
npm run seed:all           # storage bucket + ai_profiles (default = mock)
```

Then:

```bash
# A case file (one case or a list) or a directory of cases. --backend may be a
# comma-separated list to run several backends per case in one go.
node evaluation/runtime/run.mjs evaluation/runtime/cases/age-readability.mjs --backend endpoint

# Compare models on the IDENTICAL input â€” same case, two model paths:
node evaluation/runtime/run.mjs evaluation/runtime/cases/age-readability.mjs --backend endpoint,cli:claude

# Endpoint backend against a real model (seed an openrouter profile first; see
# docs/ai-configuration.md):
node evaluation/runtime/run.mjs evaluation/runtime/cases/age-readability.mjs --backend endpoint --ai-profile free

# CLI backend, offline wiring check (deterministic, no model call):
node evaluation/runtime/run.mjs evaluation/runtime/cases/age-readability.mjs --backend cli:stub

# Re-judge a stored interaction (no API calls) â€” the judge-iteration loop:
node evaluation/runtime/rejudge.mjs runs/<run_id>/<case>__<backend>/interaction.json
```

`run.mjs` flags: `--backend <spec[,spec]>`, `--ai-profile <id>` (endpoint
backend), `--judges a,b`, `--out <dir>`. Exit code is `2` if any judge fails or
errors.

## CLI backend configuration

`config/cli.example.json` defines the `claude`, `openai`, and `stub` variants
(selected with `--backend cli:<variant>`). Copy it to `config/cli.json`
(gitignored) to change the model or wrapper. Each wrapper receives the role
system prompt and the user message (`JSON.stringify({ prompt, context })`) as
file arguments and returns the model's role JSON; the backend reads `.narration`
from it. The `openai` wrapper needs the `openai` CLI and `OPENAI_API_KEY`.

## Adding things

- **New case** â€” add to `cases/<name>.mjs`. Pick a blueprint with the
  `target_age` (or other property) you want, write the fixed `given` + `action`.
- **New judge** â€” add `lib/judges/<id>.mjs` exporting `id` + `judge(...)`, then
  register it in `lib/judges/index.mjs`.
- **New action / CLI role** â€” add an entry to `ACTIONS` in `lib/roles.mjs`
  (endpoint mapping, and a `cli` mapping to also replay it locally).
