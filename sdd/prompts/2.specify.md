# SDD Phase 2: Specification Prompt

---

You are writing a specification that will be the single source of truth for implementing a feature. Future agent sessions will read this spec to understand what to build. Every decision must be captured here — nothing implicit.

## Mandatory Context Loading

Before writing the spec, read these files:

- `docs/architecture.md`
- `docs/game.md`
- `docs/project-structure.md`
- `docs/testing.md`

## Inputs

Read these files:

- Problem definition: `docs/sdd/specs/{feature-name}/problem.md`
- Research findings: `docs/sdd/specs/{feature-name}/research.md`

Pay particular attention to the "Problem Reframe" section in research.md. If research recommended changes to the problem definition, verify that problem.md has been updated before proceeding. If it hasn't, STOP and ask the human to review the reframe recommendation first.

## Instructions

Write the specification using the structure defined in `docs/sdd/templates/spec.md`.

Rules:
- Every design decision must include rationale and rejected alternatives
- Constraints must include explicit "Must Not" items — what the agent should NOT build
- Success criteria must be testable — map each to a test tier (unit, component, integration, e2e)
- The implementation checklist must have tasks that are each completable in a single agent session
- Include a "Documentation Impact" section listing which project docs need updating
- Architecture must respect the existing layer boundaries: routes → services → repositories → shared
- If this feature involves regulatory logic, the spec must enforce the rules-engine / inference-engine separation
- If this feature involves a new jurisdiction, it must be namespaced — no hardcoded country assumptions in shared code
- Where research found conflicting approaches, the spec must pick one and document why

Save as `docs/sdd/specs/{feature-name}/spec.md`.
