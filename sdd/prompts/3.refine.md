## Interview

Ask me questions — in small batches of up to FIVE — to resolve anything unclear. Focus on these categories:

1. **DATA DECISIONS**: How does existing data transition? Schema changes? Migration strategy? What happens to existing records? Does this affect the `onboarding_journeys` table or other existing tables?

2. **CONFLICT RESOLUTION**: What happens when things disagree? Multi-user edits? Local vs server state? Race conditions? Validation conflicts between client and server?

3. **PATTERN SELECTION**: Where the research found multiple valid approaches, which one did we pick? Are there remaining either/or decisions? Does this align with the patterns in `docs/architecture.md`?

4. **FAILURE & RECOVERY**: What happens when things go wrong? API timeouts? Partial failures? Crash recovery? What does the user see? What gets logged? Does the audit trail capture failure states?

5. **BOUNDARY CONDITIONS**: What are the limits? Max file sizes? Rate limits? What happens at the edges? What about empty states — zero data, first-time user, missing optional fields?

6. **PROJECT INTEGRATION**: Does this spec conflict with anything in the tech constitution? Does it respect layer boundaries? Are the quality gates sufficient? Is the testing strategy aligned with `docs/testing.md`?

After each answer I give, update your understanding of the spec. When you've run out of questions, produce an UPDATED spec with all clarifications incorporated inline (not as a separate appendix). Save it to `docs/sdd/specs/{feature-name}/spec.md`, replacing the existing version.

YOU MUST ASK QUESTIONS. Do not refine the spec indepently. Do not make assumptions. Use my answers to clarify any uncertainty and update spec accordingly.
