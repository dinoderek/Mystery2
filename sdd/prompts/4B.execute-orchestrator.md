# SDD Phase 4B: Task Execution ORCHESTRATOR Prompt

## Your task

You are the orchestrator tasked with implementing the full feature based on the phases and tasks described in `docs/sdd/specs/{feature-name}/tasks.md`.

Read: `docs/sdd/specs/{feature-name}/spec.md`
Read: `docs/sdd/specs/{feature-name}/tasks.md`

Before writing any code, read:
- `docs/architecture.md`
- `docs/game.md`
- `docs/project-structure.md`
- `docs/testing.md`

If the task involves UI work, also read:
- `docs/styling-conventions.md`
- `docs/component-inventory.md`
- `docs/screen-navigation.md`

Prepare an execution plan that maximises parallelism and spawn subagents to execute tasks. 

Feed the sub-agents with:
- Their task identifier
- The task specific prompt `sdd/prompts/4B.execute.md`
- Mandatory reading as per above
- Any other context that is relevant

Important note: E2E and Browser tests are NOT safe to run in parallel by mulitple subagents. Consider this when planning.

## Rules

1. Implement ONLY what the task plan describes. Do not refactor unrelated code. Do not add features not in the spec.
2. Follow the constraints in the spec exactly. If the spec says "must not", then do not.
3. Follow the project's layer architecture: routes are thin, business logic lives in services, data access in repositories.
4. Use Zod for runtime validation at system boundaries. TypeScript types alone are not sufficient.
5. Write tests as specified in the task. Follow the test tiers in `docs/testing.md`.
6. When the task is complete, run all applicable quality gates:
   - `npm run lint`
   - `npm run typecheck`
   - `npm run test:unit -- --run`
   - `npm run test:components -- --run` (if UI work)
   - `npm run test:stories -- --run` (if Storybook stories)
   - `npm run test:integration` (if database work, requires local Supabase)
   - `npm run test:e2e` (if end-to-end flows, requires local Supabase)
7. Update documentation listed in the task's "Doc updates" field.
8. If you encounter a blocker or ambiguity not covered by the spec, STOP and describe the issue. Do not guess.
9. After implementation, provide:
   - List of files created or modified
   - Test results from quality gates
   - Any issues the next task should be aware of
   - Documentation changes made

Begin.
