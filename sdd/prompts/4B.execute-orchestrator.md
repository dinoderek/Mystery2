# SDD Phase 4B: Task Execution ORCHESTRATOR Prompt

## Your task

You are the orchestrator tasked with implementing the full feature based on the phases and tasks described in `sdd/specs/{feature-name}/tasks.md`.

Read: `sdd/specs/{feature-name}/spec.md`
Read: `sdd/specs/{feature-name}/tasks.md`

Before writing any code, read:
- `docs/architecture.md`
- `docs/game.md`
- `docs/project-structure.md`
- `docs/testing.md`

If the task involves UI work, also read:
- `docs/styling-conventions.md`
- `docs/component-inventory.md`
- `docs/screen-navigation.md`

Before assigning work, build an execution plan from the task list metadata. For each task, identify:
- `Depends on`
- `Spec sections`
- `Input`
- `Files affected`
- `Done when`
- `Quality gates`
- `Doc updates`

Prepare an execution plan that maximises safe parallelism and spawn subagents to execute tasks.

Only run tasks in parallel when:
- all dependencies are already satisfied
- their write scopes do not overlap
- they do not run browser or E2E tests concurrently

Treat shared files such as `package.json`, shared helpers, shared test files, and shared docs as serialization points unless you deliberately assign a single task ownership of that file.

Feed the sub-agents with:
- Their task identifier
- The task specific prompt `sdd/prompts/4B.execute.md`
- The exact task section they own
- The task's referenced spec sections
- The task's `Input` files and any mandatory reading they must load
- The task's `Files affected` as their write scope
- The task's `Done when`, `Quality gates`, and `Doc updates`
- Mandatory reading as per above
- Any other context that is relevant

Important note: E2E and browser tests are NOT safe to run in parallel by multiple subagents. Consider this when planning.

## Rules

1. Implement ONLY what the task plan describes. Do not refactor unrelated code. Do not add features not in the spec.
2. Follow the constraints in the spec exactly. If the spec says "must not", then do not.
3. Write tests as specified in the task. Follow the test tiers in `docs/testing.md`.
4. The orchestrator owns dependency planning, task assignment, integration of subagent results, conflict resolution, and final regression sequencing.
5. Update documentation listed in the task's "Doc updates" field.
6. If you encounter a blocker or ambiguity not covered by the spec, STOP and describe the issue. Do not guess.
7. Subagents must not commit, start the next task, or edit outside their assigned write scope. They should return results to the orchestrator for integration.
8. When the feature is complete, run all applicable quality gates.
9. After implementation, provide:
   - List of files created or modified
   - Test results from quality gates
   - Any issues the next task should be aware of
   - Documentation changes made

Begin.
